# 协作开发指南 — 以「领养系统」为例

## 新功能涉及哪些改动

### 新合约（`PawAdoption.sol`）
- 数据结构：`AdoptionListing`（关联案例、动物信息、状态）、`Application`（申请人、留言、审核状态）
- 核心函数：`listForAdoption`、`applyToAdopt`、`approveApplicant`、`rejectApplicant`
- 与 `PawLedger` 联动：只有状态为 `CLOSED`（救助资金已到位）的案例才能发起领养

### 现有合约（`PawLedger.sol`）— 尽量少改
- 可在 `Case` 结构体中加一个 `adoptionListed` 标志，或完全解耦，视设计而定

### 前端新增
- 页面：`AdoptionBrowser`、`AdoptionDetail`、`ApplyForm`
- Hook：`useAdoption.js`
- 路由：`/adopt`、`/adopt/:id`
- 双语文案：`zh.json` + `en.json` 补充新 key
- `config.js` 更新新合约地址

### 部署脚本
- 在现有脚本中补充 `PawAdoption` 的部署步骤

## 推荐协作流程

```
1. 【定接口】
   └─ 写入 docs/xxx-spec.md，确认后再开始写代码

2. 【开分支】
   └─ git checkout -b feature/adoption-system

3. 【并行开发】
   ├─ 负责人 A：写 PawAdoption.sol + 单元测试
   └─ 负责人 B：搭前端页面、路由、i18n（先用 mock 数据）
      （B 不需要等合约部署好才能开始）

4. 【部署合约】
   └─ A 部署到 Fuji 测试网，将新地址提交到 config.js
   └─ B 将 useAdoption.js 接入真实合约

5. 【联调 + PR】
   └─ 在 Fuji 上一起测试
   └─ 发 PR → 互相 Review → 合并到 main
```

## 常见坑 & 规避原则

| 风险 | 规则 |
|---|---|
| 合约接口开发到一半改了 | **ABI 冻结后才能开始前端接线**，接口变动必须重新对齐 |
| 各自部署导致地址混乱 | **只由 A 负责部署**，地址通过 commit 写入 `config.js` |
| `config.js` 合并冲突 | B 等 A 的部署 commit 合并后再动 config |
| 新功能混入其他修复 | **新合约 = 独立 PR**，不要把领养改动和 PawLedger 的 bug fix 混在一起提交 |
