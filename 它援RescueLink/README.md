# 它援 RescueLink

**它援 RescueLink 是一个让流浪动物救助更简单的需求连接与可信记录工具。**

它以单个救助个案为核心，把原本分散在小红书、微信、群聊、私聊和截图里的信息，整理成一个**可查看、可更新、可分享、可持续记录**的统一页面，让救助需求更容易被看见、被理解、被承接，降低求助门槛与信任门槛，也让救助支持者、医院、寄养方和潜在领养人等可帮助资源，能够更快、更准确地接入同一个 case。

---

## 核心功能

### 1. 个案创建与展示
- 创建救助个案
- 填写动物情况、地点、紧急程度和需求类型
- 生成统一的 case 页面
- 对外展示当前状态、需求和进展

### 2. 需求连接
- 将救助需求结构化展示为具体需求项
- 展示当前最需要的帮助类型
- 展示需求承接状态
- 帮助外部资源快速判断自己可以帮哪一部分

### 3. 进展更新
- 持续更新当前状态
- 展示已完成事项与待完成事项
- 记录 case 时间线
- 减少发起人重复解释和重复同步的负担

### 4. 凭证与记录
- 上传医疗单据、检查报告、付款记录、签收记录等材料
- 沉淀关键进展对应的补充信息
- 让救助记录更完整、更容易回看

### 5. 传播与分享
- 生成对外可分享页面
- 自动生成结构化传播模板
- 帮助发起人在发帖前补齐关键信息
- 提高求助内容在社交平台上的可理解性和传播效率

### 6. 链上存证
- 对 case 创建、状态更新、凭证上传、关键节点和结案等信息进行摘要存证
- 保留链下可用性，同时增强关键记录的可追溯性

## 项目简介

当前流浪动物救助高度依赖民间力量。一次真实的救助，往往需要发现人、当前责任人、志愿者、医院、寄养方、支持者和领养人等多方临时接力完成。

但现实中常见的问题是：

- 信息分散，难以统一查看
- 需求表达不清，外部帮助者不知道该怎么接入
- 当前状态和缺口不透明，容易造成重复沟通
- 票据、报告、签收记录等重要材料容易丢失
- 发起人需要反复解释、更新、协调，负担很重
- 求助内容在传播前就已经信息缺失，影响搜索、转发和匹配效率

**它援 RescueLink 想解决的，不只是“资源不够”，更是资源连接困难、信息结构混乱、记录脆弱和信任成本高的问题。**

- 需求结构化展示

## 为什么帮助者愿意使用它

对于志愿者、寄养方、物资支持者、转运协助者和潜在领养人来说，它援 RescueLink 的价值不只是“看一个求助帖”，而是：

- 能更快看懂一个 case 现在到底缺什么
- 能判断这件事是否还需要自己接入，减少重复沟通
- 能直接找到自己能帮的那一部分，而不是从零开始问情况
- 能看到最新进展、当前责任人和关键记录，降低信息不透明带来的顾虑
- 能让自己的帮助被记录下来，减少帮助之后信息断掉或反复确认的成本

也就是说，它不是让帮助者“额外做一件事”，而是让他们**更低成本地接入帮助**。

## 核心场景

### 1. 个人救助型个案

围绕单个动物、单次事件发起的救助需求页面。

典型需求包括：

- 紧急送医
- 检查与治疗
- 临时寄养
- 转运协调
- 物资支持
- 志愿协助
- 领养扩散
- TNR
- 后续回访

### 2. 组织资源求助型 Case

围绕组织、救助小院或长期救助主体发起的持续性资源需求页面。

典型需求包括：

- 猫粮 / 狗粮 / 猫砂
- 药品与医疗耗材
- 长期寄养位
- 志愿者招募
- 运输资源
- 固定合作医院支持
- 月度运营型支持

## 产品逻辑

它援 RescueLink 的核心逻辑很简单：

### 1. 有人发起一个真实的 rescue case
发起人填写动物情况、当前状态、地点、紧急程度和需要的帮助类型。

### 2. 平台把需求结构化展示出来
把“我需要帮忙”拆成更清楚的需求项，例如：

- 送医
- 寄养
- 物资
- 医疗支持
- 志愿协助
- 领养
- TNR
- 组织补给

### 3. 外部帮助者快速接入
帮助者可以快速判断：

- 这个 case 是什么情况
- 当前最缺什么
- 哪些需求已经有人承接
- 自己可以帮哪一部分

### 4. 当前责任人持续更新进展
平台记录：

- 当前状态
- 已完成事项
- 待完成事项
- 需求承接状态
- 票据、报告、签收等补充材料
- 关键节点记录

### 5. 形成一个可持续更新、可分享、可追溯的需求页面
让一个原本碎片化、难以承接的求助，变成一个更容易连接资源的 case 页面。

## 产品流程

发现动物  
→ 发起救助个案  
→ 填写基础信息与紧急情况  
→ 上传现场图片 / 视频  
→ 发布需求  
→ 平台生成统一个案页面  
→ 自动生成社交传播模板  
→ 外部参与者查看并提供帮助  
→ 当前责任人持续更新进展  
→ 上传票据、报告、签收等凭证  
→ 记录关键节点  
→ 完成寄养 / 领养 / 结案  
→ 对关键记录进行摘要存证

## 角色设计

### 发起人
创建 case 并提交初始信息的人，默认是个案初始责任人。

### 当前责任人
当前对个案推进承担主要责任的人，负责持续更新进展和维护 case 状态。

### 帮助者
围绕某个具体需求提供帮助的人，包括志愿者、寄养方、支持者、转运协助者和潜在领养人等。

### 服务方
提供医疗、寄养、运输等服务的机构或个人。

### 组织方
发起组织资源求助型 case 的民间救助组织或救助小院。

## 资金与支持逻辑

它援 RescueLink **不是一个直接打款给个人的募捐平台**。

我们更关注的是：

- case 级别的支持记录
- 与具体需求绑定的支持行为
- 更清楚的资源匹配
- 可追溯的支持过程

我们的原则是：

- 支持应绑定具体 case
- 支持应对应明确用途
- 支持对象应尽量清晰
- 支持过程应可记录、可追溯
- 关键记录可进行链上摘要存证

在当前版本中，我们**不做完整支付或托管系统**，而是优先验证以下能力：

- 展示明确需求
- 展示支持类型
- 展示承接状态
- 上传相关凭证
- 记录完成状态
- 对关键节点进行摘要存证

---

## 为什么使用区块链

我们不是把所有数据都放到链上，而是只把**关键节点的摘要信息**写到链上。

因为流浪动物救助具有这些特点：

- 多方临时接力
- 责任链条分散
- 信任成本高
- 关键节点需要留痕
- 凭证需要可追溯和防篡改基础

我们的原则是：

**链下可用，链上可信。**
